# B5-1: SQL로 만드는 나만의 데이터베이스

## 1. 프로젝트 개요

이 프로젝트는 백엔드 프레임워크 없이 SQL만 사용하여 직접 데이터베이스를 설계하고, 데이터를 입력한 뒤, 다양한 SQL 쿼리를 통해 필요한 정보를 조회하는 실습 프로젝트이다.

주제는 **카페 주문 관리 데이터베이스**로 설정하였다. 카페에서 발생하는 고객 정보, 메뉴 정보, 주문 정보, 주문 상세 정보를 각각의 테이블로 나누어 저장하고, 테이블 간의 관계를 PK와 FK로 연결하였다.

이를 통해 단순히 데이터를 한 테이블에 저장하는 것이 아니라, 여러 테이블로 나누어 구조화하고 JOIN, GROUP BY, 서브쿼리 등을 사용해 실제 서비스에서 필요한 데이터를 조회하는 흐름을 실습하였다.

---

## 2. 사용한 개발 환경

* DB: SQLite
* 실행 방식: SQLite CLI
* 제출명: B5-1
* 사용 파일:

  * `schema.sql`
  * `insert_data.sql`
  * `queries.sql`
  * `cafe_order.db`
  * `results/`

SQLite를 선택한 이유는 별도의 서버 실행이 필요하지 않고, 하나의 데이터베이스 파일로 관리할 수 있어 SQL 입문 실습에 적합하기 때문이다.

---

## 3. 데이터베이스 주제

이 프로젝트의 데이터베이스 주제는 **카페 주문 관리 시스템**이다.

카페 운영 상황을 기준으로 다음과 같은 데이터를 관리하도록 설계하였다.

* 고객 정보
* 메뉴 정보
* 주문 정보
* 주문 상세 정보

예를 들어 한 명의 고객은 여러 번 주문할 수 있고, 하나의 주문에는 여러 메뉴가 포함될 수 있다. 또한 하나의 메뉴는 여러 주문 상세 항목에서 반복적으로 사용될 수 있다. 이러한 관계를 표현하기 위해 테이블을 분리하고 외래키를 사용하였다.

---

## 4. 테이블 구조

총 4개의 테이블을 생성하였다.

### 4.1 customers

고객 정보를 저장하는 테이블이다.

| 컬럼명        | 설명       |
| ---------- | -------- |
| id         | 고객 고유 ID |
| name       | 고객 이름    |
| email      | 고객 이메일   |
| phone      | 고객 전화번호  |
| created_at | 고객 등록일   |

`id`는 기본키로 설정하였고, `email`은 중복될 수 없도록 UNIQUE 제약조건을 적용하였다.

---

### 4.2 menu_items

카페 메뉴 정보를 저장하는 테이블이다.

| 컬럼명          | 설명       |
| ------------ | -------- |
| id           | 메뉴 고유 ID |
| name         | 메뉴 이름    |
| category     | 메뉴 카테고리  |
| price        | 메뉴 가격    |
| is_available | 판매 가능 여부 |

`id`는 기본키로 설정하였고, `name`은 중복되지 않도록 UNIQUE 제약조건을 적용하였다. 또한 `price`는 0보다 큰 값만 입력되도록 CHECK 제약조건을 설정하였다.

---

### 4.3 orders

고객의 주문 정보를 저장하는 테이블이다.

| 컬럼명         | 설명        |
| ----------- | --------- |
| id          | 주문 고유 ID  |
| customer_id | 주문한 고객 ID |
| order_date  | 주문 날짜     |
| status      | 주문 상태     |

`customer_id`는 `customers` 테이블의 `id`를 참조하는 외래키이다. 이를 통해 어떤 고객이 어떤 주문을 했는지 연결할 수 있다.

---

### 4.4 order_items

주문 안에 포함된 메뉴 상세 정보를 저장하는 테이블이다.

| 컬럼명          | 설명          |
| ------------ | ----------- |
| id           | 주문 상세 고유 ID |
| order_id     | 주문 ID       |
| menu_item_id | 메뉴 ID       |
| quantity     | 수량          |

`order_id`는 `orders` 테이블의 `id`를 참조하고, `menu_item_id`는 `menu_items` 테이블의 `id`를 참조한다. 이를 통해 하나의 주문에 여러 개의 메뉴가 포함될 수 있도록 설계하였다.

---

## 5. 테이블 관계

이 프로젝트에서는 다음과 같은 관계를 설정하였다.

```text
customers.id → orders.customer_id
orders.id → order_items.order_id
menu_items.id → order_items.menu_item_id
```

이를 통해 다음과 같은 1:N 관계가 만들어진다.

1. 한 명의 고객은 여러 개의 주문을 할 수 있다.
2. 하나의 주문은 여러 개의 주문 상세 항목을 가질 수 있다.
3. 하나의 메뉴는 여러 주문 상세 항목에 포함될 수 있다.

이처럼 데이터를 여러 테이블로 나누어 저장하면 중복을 줄일 수 있고, 데이터 간의 관계를 명확하게 관리할 수 있다.

---

## 6. 적용한 제약조건

데이터의 무결성을 유지하기 위해 다음과 같은 제약조건을 적용하였다.

* `PRIMARY KEY`: 각 테이블의 행을 고유하게 식별하기 위해 사용하였다.
* `FOREIGN KEY`: 테이블 간 관계를 연결하기 위해 사용하였다.
* `NOT NULL`: 반드시 값이 필요한 컬럼에 적용하였다.
* `UNIQUE`: 중복되면 안 되는 이메일과 메뉴 이름에 적용하였다.
* `CHECK`: 가격과 수량이 0보다 큰 값만 입력되도록 설정하였다.

특히 외래키를 사용하여 존재하지 않는 고객 ID나 메뉴 ID가 주문 데이터에 들어가는 것을 방지하였다.

---

## 7. 샘플 데이터

각 테이블에는 최소 10행 이상의 샘플 데이터를 입력하였다.

* `customers`: 고객 10명
* `menu_items`: 메뉴 10개
* `orders`: 주문 10개
* `order_items`: 주문 상세 20개

데이터 입력 시에는 외래키 관계를 고려하여 부모 테이블의 데이터를 먼저 입력하였다.

입력 순서는 다음과 같다.

```text
1. customers
2. menu_items
3. orders
4. order_items
```

`orders` 테이블은 `customers` 테이블을 참조하고, `order_items` 테이블은 `orders`와 `menu_items` 테이블을 참조하기 때문에 이 순서대로 데이터를 입력해야 한다.

## 8. 실행 방법

먼저 프로젝트 폴더에서 SQLite 데이터베이스 파일을 생성하고, 스키마와 데이터를 적용한다.

```bash
sqlite3 cafe_order.db < schema.sql
sqlite3 cafe_order.db < insert_data.sql
```

데이터베이스에 접속하려면 다음 명령어를 사용한다.

```bash
sqlite3 cafe_order.db
```

SQLite 내부에서 결과를 보기 좋게 출력하기 위해 다음 명령어를 실행한다.

```sql
.headers on
.mode column
```

테이블 목록은 다음 명령어로 확인할 수 있다.

```sql
.tables
```

SQLite를 종료할 때는 다음 명령어를 사용한다.

```sql
.quit
```

## 9. 핵심 SQL 쿼리

`queries.sql` 파일에는 총 15개의 핵심 SQL 쿼리를 작성하였다.

포함한 쿼리 유형은 다음과 같다.

* 기본 조회
* WHERE 조건 검색
* ORDER BY 정렬
* LIMIT 제한
* INNER JOIN
* LEFT JOIN
* COUNT 집계
* SUM 집계
* AVG 집계
* GROUP BY
* 서브쿼리
* UPDATE
* DELETE

각 쿼리는 단순히 데이터를 조회하는 것뿐만 아니라, 실제 카페 주문 데이터에서 필요한 정보를 확인할 수 있도록 작성하였다.

예를 들어 다음과 같은 내용을 SQL로 확인하였다.

* 전체 고객 목록 조회
* 특정 가격 이상인 메뉴 조회
* 메뉴 가격순 정렬
* 가장 저렴한 메뉴 3개 조회
* 주문 정보와 고객 정보 함께 조회
* 주문 상세와 메뉴명 함께 조회
* 고객별 주문 목록 조회
* 모든 고객과 주문 여부 조회
* 전체 주문 수 계산
* 주문 상태별 주문 수 집계
* 카테고리별 평균 메뉴 가격 계산
* 주문별 총 결제 금액 계산
* 평균 가격보다 비싼 메뉴 조회
* 특정 메뉴 가격 수정
* 취소된 주문 상세 데이터 삭제

## 10. 인덱스 적용

`orders.customer_id` 컬럼에 인덱스를 생성하였다.

```sql
CREATE INDEX idx_orders_customer_id ON orders(customer_id);
```

이 컬럼은 고객별 주문을 조회하거나 `customers` 테이블과 `orders` 테이블을 JOIN할 때 자주 사용된다. 따라서 인덱스를 생성하면 특정 고객의 주문을 찾거나 테이블을 연결할 때 검색 성능을 향상시키는 데 도움이 된다.

## 12. 프로젝트를 통해 배운 점

이번 프로젝트를 통해 데이터베이스가 단순히 데이터를 저장하는 공간이 아니라, 데이터 간의 관계와 규칙을 함께 관리하는 구조라는 것을 이해할 수 있었다.

특히 PK는 각 데이터를 고유하게 구분하기 위해 사용되고, FK는 서로 다른 테이블의 데이터를 연결하기 위해 사용된다는 점을 실습을 통해 확인하였다. 또한 1:N 관계를 사용하면 한 명의 고객이 여러 주문을 가질 수 있고, 하나의 주문이 여러 메뉴를 포함할 수 있다는 실제 서비스 구조를 표현할 수 있다는 것을 알게 되었다.

SQL 쿼리 작성 과정에서는 SELECT, INSERT, UPDATE, DELETE의 역할을 구분할 수 있었고, JOIN을 사용하면 여러 테이블에 나누어 저장된 데이터를 하나의 결과로 조회할 수 있다는 점을 배웠다. 또한 GROUP BY와 COUNT, SUM, AVG를 사용하여 데이터를 단순 조회하는 것에서 나아가 집계와 분석도 가능하다는 것을 확인하였다.

이번 실습을 통해 이후 JPA나 ORM을 배울 때도 내부적으로 테이블 관계와 SQL이 어떻게 동작하는지 더 쉽게 이해할 수 있는 기초를 만들 수 있었다.

## 14. 마무리

이번 프로젝트에서는 카페 주문 관리라는 주제를 바탕으로 직접 데이터베이스를 설계하고 SQL을 작성하였다. 테이블을 나누어 데이터를 저장하고, PK와 FK를 통해 관계를 설정했으며, 15개의 핵심 SQL 쿼리를 통해 데이터 조회, 조인, 집계, 수정, 삭제를 실습하였다.

이를 통해 엑셀처럼 단순히 데이터를 나열하는 방식과 달리, 데이터베이스는 관계와 제약조건을 통해 데이터를 더 안전하고 체계적으로 관리할 수 있다는 점을 이해할 수 있었다.
